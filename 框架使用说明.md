#  二   环境配置说明

打开工程后先确认这些东西已经装好：TI MSPM0 SDK、CCS或SysConfig、Arm GNU Toolchain、CMake、Ninja。要烧录的话再确认OpenOCD和CMSIS-DAP/DAPLink驱动。

## VS Code/CMake里怎么选

1. 打开工程根目录 `MSPM0G3507_BASIC`。
2. 如果CMake Tools让你选Kit，不要选MSVC、MinGW这类电脑本机编译器；本工程要用 `arm-none-eabi-gcc`。
3. 确认Arm GNU Toolchain的 `bin` 已经加入系统PATH，终端里能直接运行 `arm-none-eabi-gcc --version`。
4. 本工程默认工具链文件是 `cmake/arm-none-eabi-gcc.cmake`，正常不需要手动改。

## 路径在哪里改

不同电脑最容易不一样的是这四个路径，都在 `cmake/mspm0g3507.cmake` 文件开头：

- `MSPM0_SDK_DIR`：MSPM0 SDK根目录。
- `SYSCONFIG_CLI`：`sysconfig_cli.bat` 的位置。
- `OPENOCD_EXE`：`openocd.exe` 的位置。
- `OPENOCD_SCRIPTS_DIR`：OpenOCD的 `scripts` 目录。



## 改路径后的注意事项

- CMake会缓存路径；换电脑、移动工程、升级SDK后，最好删掉旧 `build` 目录或重新配置。
- `Required MSPM0 toolchain path not found` 基本就是SDK、SysConfig或OpenOCD路径没对。
- `Could not find arm-none-eabi-gcc` 基本就是Arm GNU Toolchain没进PATH。
- 编译通过但烧录失败，优先查OpenOCD路径、探针连接、驱动和板子供电。
- 不要手改 `build/syscfg` 里的生成文件；引脚和外设配置改 `empty.syscfg`。

## BSP兼容层说明

这次移植BSP时，原则是保留跃鹿框架原本给module层使用的函数名、结构体名和字段语义，不直接改module层的调用方式。比如 `GPIOInstance`、`PWMInstance`、`USARTInstance`、`SPIInstance`、`IICInstance`，以及 `GPIORegister()`、`PWMRegister()`、`USARTSend()`、`SPITransRecv()` 这些接口都保持原名。

具体做法是在 `bsp` 根目录新增了一层 TI/MSPM0 兼容文件：

- `bsp_mspm0_compat.h/.c`：把原来STM32风格的 `GPIO_TypeDef`、`GPIO_PinState`、`UART_HandleTypeDef`、`SPI_HandleTypeDef`、`I2C_HandleTypeDef`、`TIM_HandleTypeDef` 等名字映射到MSPM0 DriverLib对应的外设寄存器和轻量handle。
- `main.h`、`gpio.h`、`tim.h`、`usart.h`、`spi.h`、`i2c.h`：这些不是STM32生成文件，而是为了兼容原BSP include习惯写的TI版本入口。
- `cmsis_os.h`：提供CMSIS-RTOS v1风格的最小兼容接口，FreeRTOS构建时转到原生FreeRTOS任务/通知，NoRTOS构建时提供空实现，保证备用构建不炸。

外设配置不再参考CubeMX，而是以SysConfig生成的 `ti_msp_dl_config.h/.c` 为准。比如当前工程里已经由SysConfig生成了 `PWM_0~PWM_3`、`I2C_0`、`UART_0`、`SPI_0`、`ADC12_0` 和 `GPIO_GRP_0` 等实例，BSP内部实现应优先使用这些宏和 DriverLib API。

目前 `GPIO/PWM/SPI/IIC/USART/Flash/log/tools/init` 已经进入CMake编译并通过。需要注意的是，`SPI/IIC/USART` 的 `IT/DMA` 枚举和函数入口被保留下来了，但当前内部主要先按阻塞方式兜底，因为SysConfig还没有给这些通信外设完整配置DMA通道和中断接收链路。后续如果module确实要用DMA、Idle接收或中断回调，需要再按具体外设补SysConfig配置和ISR逻辑。

Flash也已经从STM32的 `0x08000000` 地址系改成MSPM0G3507当前链接脚本对应的 `0x00000000 ~ 0x00020000`。真正用于保存参数前，建议单独约定一块参数区，避免擦写到程序正文。

## 当前工程迁移注意点

### FreeRTOS时钟和tick

当前SDK里的 `FreeRTOSConfig.h` 使用：

```c
#define configCPU_CLOCK_HZ    ( ( unsigned long ) 32000000 )
#define configTICK_RATE_HZ    1000
```

当前SysConfig生成的 `build/syscfg/ti_msp_dl_config.h` 里 `CPUCLK_FREQ` 也是 `32000000`，所以现在两边是对得上的，`osDelay(1)` 基本就是1ms。

后续如果在 `empty.syscfg` 里改了SYSCTL时钟，务必重新看生成文件里的 `CPUCLK_FREQ`，并确认它和FreeRTOS的 `configCPU_CLOCK_HZ` 一致。否则 `vTaskDelay()`、`osDelay()`、软件定时器、任务调度节拍都会不准，表现可能是“任务周期怪怪的”，而不是编译报错。

现在CMake用的是SDK里的FreeRTOS配置目录：

```cmake
MSPM0_FREERTOS_CONFIG_DIR =
    ${MSPM0_SDK_DIR}/kernel/freertos/builds/LP_MSPM0G3507/release
```

如果以后要长期使用非32MHz主频，更推荐复制一份本工程自己的 `FreeRTOSConfig.h`，再改 `cmake/mspm0g3507.cmake` 指到本地配置，不要直接改TI SDK安装目录。

### FreeRTOS堆和任务栈

SDK这份配置里 `configTOTAL_HEAP_SIZE` 只有3KB。当前 `cmsis_os.h` 的 `osThreadCreate()` 底层用的是 `xTaskCreate()`，也就是用户任务的TCB和栈会从FreeRTOS heap里分配。

这意味着后续打开更多任务时，可能出现“能编译、能进main、但任务没创建成功”的情况。尤其注意：

- `osThreadDef(..., stacksz)` 里的栈大小按FreeRTOS习惯是以 `StackType_t` 为单位，不是字节；在Cortex-M0+上通常1个单位是4字节。
- `robot_task.h` 里如果给任务写 `1024`，实际大约就是4KB栈，已经超过当前3KB heap。
- `app/freertos_hooks.c` 只给Idle Task和Timer Task提供静态内存，不代表用户任务也是静态创建。

后续比较稳的做法有两个：要么把 `configTOTAL_HEAP_SIZE` 调大，并检查 `osThreadCreate()` 的返回值；要么把常驻任务逐步改成 `xTaskCreateStatic()` / CMSIS静态创建方式。

### Tickless idle和低功耗

SDK配置里 `configUSE_TICKLESS_IDLE` 当前是1。SysConfig构建时也会提示一些外设在STOP/STANDBY模式下不保留寄存器内容。做电机控制、通信接收、PWM输出这类实时任务时，如果遇到“运行一段时间后外设状态不对”或“低功耗后外设没恢复”，需要优先检查FreeRTOS低功耗策略和SysConfig的外设retention说明。

比赛代码如果暂时不做低功耗，建议先不要主动进STOP/STANDBY；如果要关tickless idle，也应通过本地 `FreeRTOSConfig.h` 统一处理。

### DMA和中断接收目前不是完整链路

当前SysConfig里虽然加入了DMA模块，生成文件里也有 `SYSCFG_DL_DMA_init()`，但UART/SPI/I2C还没有按具体外设完整配置DMA通道、触发源、NVIC中断和回调处理。

BSP里 `USART/SPI/IIC` 的 `IT/DMA` 枚举和函数入口是为了兼容原跃鹿module层先保留下来的，目前主要还是阻塞式兜底。后续如果某个模块真的要用DMA或中断接收，不能只把模式枚举改成DMA，还需要：

- 在 `empty.syscfg` 里给对应外设配置DMA通道和trigger。
- 重新生成后确认 `ti_msp_dl_config.h` 里的DMA宏名、IRQ名和外设实例名。
- 在BSP里补对应ISR、回调、busy状态处理和缓存区生命周期。
- 用逻辑分析仪或串口工具实际验证收发时序。

### SysConfig生成文件不要手改

`build/syscfg/ti_msp_dl_config.h`、`build/syscfg/ti_msp_dl_config.c`、`device_linker.lds`、`device.opt` 都是生成物。改引脚、外设实例、波特率、PWM频率、ADC通道、DMA配置时，改 `empty.syscfg`，然后重新构建。

代码里不要猜宏名，先看生成的 `ti_msp_dl_config.h`。比如当前工程实际生成的是 `UART_0`、`SPI_0`、`I2C_0`、`PWM_0~PWM_3`、`TIMER_0` 等名字。

### DWT和时间测量

STM32上常见的DWT CYCCNT在MSPM0G3507的Cortex-M0+上不能照搬。当前 `bsp/dwt` 只是做了MSPM0友好的兼容：短延时用 `delay_cycles`，调度器启动后的时间线主要依赖FreeRTOS tick。

如果后续要做高精度运行时间统计、微秒级调度或控制环耗时测量，建议单独用SysConfig配置一个硬件Timer做时间基准，而不是继续按STM32的DWT思路写。

### printf和日志输出

`app/syscalls_stub.c` 目前只是让newlib裸机工程能链接，里面的 `_write()` 会假装写成功，但不会真的把字符发到UART/RTT/USB。因此 `printf()` 现在不是可见串口输出。

如果要把日志接到串口或RTT，需要改 `_write()` 或 `bsp_log` 的底层输出，并同时确认SysConfig里的UART波特率、TX引脚和上位机串口参数。
